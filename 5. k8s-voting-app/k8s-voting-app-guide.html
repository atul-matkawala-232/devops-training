Microservices Architecture

Before deploying, let's understand what microservices are, why they exist, and how the Docker Voting App demonstrates this architecture perfectly.

5 microservices Docker official sample Kubernetes deployment Docker Desktop · macOS
🏗️
What is Microservices Architecture?
The shift from monolith to independent services

In the old world, an application was one giant codebase — a monolith. Deploy one thing, it runs everything. Simple to start, painful to scale.

🧱 Monolith
  • ✗ One huge codebase
  • ✗ Scale entire app for 1 bottleneck
  • ✗ One bug can crash everything
  • ✗ All teams deploy together
  • ✗ Locked into one tech stack
🧩 Microservices
  • ✓ Small, focused services
  • ✓ Scale only what's needed
  • ✓ Failures are isolated
  • ✓ Teams deploy independently
  • ✓ Each service picks its own tech
💡 The Core Principle Each microservice does one thing well, runs in its own process, communicates over a network (HTTP or message queues), and can be deployed, scaled, and updated independently. Think of it like a kitchen: the chef, sommelier, and host each have a specialized role — not one person doing everything.
🎯
Why Kubernetes is Perfect for Microservices
The platform built for this problem
🔁 Self-healingIf a service crashes, Kubernetes restarts it — other services keep running.
📈 Independent scalingGetting more votes? Scale only the voting service — not the result app.
🌐 Service discoveryServices find each other by name — no hardcoded IPs needed.
🚀 Rolling updatesUpdate one service with zero downtime while others stay live.

Step-by-Step Vote Journey
What happens when you click "Cats"?
1
👤 User visits localhost:30004 and clicks "Cats"

Browser sends an HTTP POST request to the Voting App (Python/Flask). Kubernetes NodePort routes external port 30004 into the cluster to the voting-app pod.

2
🗳️ Voting App pushes vote to Redis

The Flask app connects to redis:6379 (ClusterIP Service name — DNS-based service discovery!) and pushes the vote as a list item. Redis acts as a fast message queue.

3
⚙️ Worker reads from Redis, writes to PostgreSQL

The .NET Worker continuously polls Redis for new votes. For each vote it finds, it UPSERTs (insert or update) a row in PostgreSQL at db:5432. This is the only way data moves from Redis to Postgres.

4
📊 Result App reads PostgreSQL and displays live results

The Node.js Result App connects to db:5432 and queries vote counts every second. It serves a real-time dashboard at port 30005 using WebSockets for live updates without page refresh.

🔑 Kubernetes Service Discovery in Action Notice the Voting App connects to "redis" — not an IP address. The Worker connects to "db" — not an IP. Kubernetes DNS automatically resolves these Service names to the correct ClusterIP. This is how microservices find each other without hardcoding IPs.

Deployment · Step 2

Deploy Redis

Redis is the in-memory message queue. The Voting App writes votes here, the Worker reads them. It needs a ClusterIP Service so other pods can reach it by the name "redis".

⚡
Redis Deployment + Service
2 YAML files
redis-deploy.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: redis-deploy
  namespace: vote
  labels:
    app: redis
spec:
  replicas: 1
  selector:
    matchLabels:
      app: redis
  template:
    metadata:
      labels:
        app: redis
    spec:
      containers:
      - name: redis
        image: redis:5.0-alpine
        ports:
        - containerPort: 6379
redis-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: redis          # Voting App connects to "redis" — this is that name
  namespace: vote
  labels:
    app: redis
spec:
  type: ClusterIP       # Internal only — pods only, no browser access
  selector:
    app: redis
  ports:
  - port: 6379
    targetPort: 6379
# Save both files then apply
kubectl apply -f redis-deploy.yaml
kubectl apply -f redis-service.yaml

# Verify
kubectl get deploy redis-deploy
kubectl get svc redis
kubectl get pods -l app=redis
NAME READY UP-TO-DATE AVAILABLE
redis-deploy 1/1 1 1

NAME TYPE CLUSTER-IP PORT(S)

Deployment · Step 3

Deploy PostgreSQL

PostgreSQL is the persistent database. The Worker writes vote tallies here. The Result App reads from here. Its Service name must be "db" — that's what both apps expect.

🐘
PostgreSQL Deployment + Service
postgres-deploy.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgres-deploy
  namespace: vote
  labels:
    app: db
spec:
  replicas: 1
  selector:
    matchLabels:
      app: db
  template:
    metadata:
      labels:
        app: db
    spec:
      containers:
      - name: postgres
        image: postgres:9.4
        env:
        - name: POSTGRES_USER
          value: "postgres"
        - name: POSTGRES_PASSWORD
          value: "postgres"
        ports:
        - containerPort: 5432
postgres-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: db              # Worker and Result App connect to "db"
  namespace: vote
  labels:
    app: db
spec:
  type: ClusterIP
  selector:
    app: db
  ports:
  - port: 5432
    targetPort: 5432

kubectl apply -f postgres-service.yaml

Deployment · Step 5

Deploy the Worker

The .NET background service that bridges Redis and PostgreSQL. It needs no Service — it reaches out to other services, nothing reaches in to it. This is the glue of the whole pipeline.

⚙️
Worker Deployment — No Service Needed
⚙️ Why no Service for Worker? The Worker is a background consumer — it never receives inbound connections. It only makes outbound connections to Redis and PostgreSQL. Therefore it needs no Kubernetes Service at all. It just runs silently, processing votes.
worker-deploy.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: worker-deploy
  namespace: vote
  labels:
    app: worker
spec:
  replicas: 1
  selector:
    matchLabels:
      app: worker
  template:
    metadata:
      labels:
        app: worker
    spec:
      containers:
      - name: worker
        image: dockersamples/examplevotingapp_worker
        # No ports — worker receives no inbound connections
kubectl apply -f worker-deploy.yaml

kubectl get deploy worker-deploy
kubectl get pods -l app=worker

# Watch the worker logs to see it processing votes
kubectl logs -l app=worker -f
NAME READY UP-TO-DATE AVAILABLE
worker-deploy 1/1 1 1

# Logs will show:
Connecting to db
Found redis at 10.96.x.x
Connecting to db
✅ Now go cast a vote at localhost:30004!After clicking Cats or Dogs, check the Worker logs — you should see it processing your vote in real time.
