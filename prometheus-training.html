The Three Pillars of Observability

Before diving into Prometheus, it's critical to understand why we need observability at all. When you deploy applications on servers or in the cloud, things go wrong — memory fills up, a service crashes, a database slows down. How do you even know something is wrong, and how do you figure out why?

The answer lies in observability — the ability to understand what is happening inside your systems from the outside. The industry has converged on three complementary signals: Metrics, Logs, and Traces.

Think of running a hospital. You have vital signs (metrics) — a patient's heart rate, blood pressure, and temperature measured every few seconds, giving you a numeric overview of their state. You have a medical chart (logs) — a chronological record of every event, nurse note, and medication administered. And you have a diagnostic trace (traces) — a step-by-step record showing exactly which organs and systems were involved when a problem occurred. Each one tells you a different part of the story.

📊

Metrics

Numeric measurements of your system collected over time. They tell you how much and how often — but not why.

CPU % Request Rate Error Count Latency (ms) Memory bytes
📜

Logs

Timestamped records of discrete events that happened in your system. They tell you what happened — in rich, human-readable detail.

Error messages Access logs Audit trails Debug output
🔍

Traces

A record of the path a request takes through distributed services. They tell you where time was spent across a whole system.

Distributed spans Service maps Latency breakdown Jaeger / Zipkin

How the Three Pillars Work Together

In practice, you use all three together. A metric alert fires: "Error rate just spiked to 15%". You open logs and see: "NullPointerException in OrderService at 14:32:05". You pull a trace and discover: "The request went through API Gateway → OrderService → PaymentService, and the failure was inside PaymentService on line 247 after 3 retries."

📌 Where Does Prometheus Fit?

Prometheus is a metrics system. It collects, stores, and queries numeric time-series data. It is not a logging tool or tracing tool — though it integrates seamlessly with tools like Grafana Loki (logs) and Grafana Tempo (traces) for full observability.

02

Introduction to Prometheus

Prometheus is an open-source monitoring and alerting toolkit originally built at SoundCloud in 2012 and donated to the Cloud Native Computing Foundation (CNCF) in 2016. It is now one of the most widely adopted monitoring systems in the world.

Think of Prometheus as a highly disciplined data journalist. Every 15 seconds (by default), it visits every service it is configured to monitor, collects numerical data, and records it with a timestamp. Over time, it builds up a rich history of how every number in your system has changed — giving you the ability to answer questions like "what was the average CPU usage across all web servers between 2pm and 3pm last Tuesday?"

Key Characteristics

Pull-Based Model

Prometheus actively "scrapes" (polls) targets. Your applications expose an HTTP endpoint, and Prometheus visits it on a schedule. This is the PULL model — the opposite of most traditional monitoring where apps PUSH data out.

Time-Series Database

Every metric is stored as a series of (timestamp, value) pairs, identified by a metric name and key-value labels. This lets you slice and dice data in powerful ways.

PromQL Query Language

Prometheus has its own powerful query language (PromQL) for filtering, aggregating, and calculating rates across your time-series data.

Standalone & Self-Sufficient

Prometheus stores its own data locally. It doesn't need an external database or cloud service — making it easy to get started fast.

The Data Model: Understanding Time-Series

Every piece of data in Prometheus is a time-series: a sequence of float64 values, each paired with a millisecond-precision timestamp. Each time-series is uniquely identified by a metric name plus a set of key=value labels.

PROMETHEUS DATA MODEL metric_name{label1="value1", label2="value2"} 
# Format: metric_name{labels} value timestamp

# HTTP requests total for a specific handler and method
http_requests_total{method="GET", handler="/api/users", status="200"}  1547  1720000000000
http_requests_total{method="POST", handler="/api/users", status="201"}   302  1720000000000
http_requests_total{method="GET", handler="/api/users", status="500"}    12  1720000000000

# Each unique combination of labels = a separate time-series
# Labels let you filter: "show me only 500 errors on GET /api/users"

The power of labels is immense. Instead of creating separate metrics like http_get_errors and http_post_errors, you create one metric http_requests_total and use labels to differentiate. PromQL can then aggregate across any dimension.

03

Why Prometheus? The Need for Modern Monitoring

Before tools like Prometheus, the dominant monitoring approach was systems like Nagios, Zabbix, or SNMP — built for a world of static servers with fixed IP addresses. The cloud-native world is radically different.

Old monitoring was like a stationary checkpoint on a highway — great for monitoring a fixed road with predictable traffic. But modern cloud infrastructure is like a city where new roads appear and disappear every minute, hundreds of cars travel on each, and the map changes constantly. You need a dynamic monitoring system that discovers and monitors targets automatically — that's Prometheus with service discovery.

Problems with Traditional Monitoring

  • Static configuration: You had to manually register every host. Containers that spin up and die in seconds are impossible to track.
  • Push overload: Every monitored agent pushed data to a central server — as systems scaled, the central server became a bottleneck and single point of failure.
  • Coarse data: Metrics were often sampled every 5–10 minutes, making short-lived spikes invisible.
  • Poor multi-dimensionality: It was hard to ask questions like "show me error rates, grouped by region, for services running version 2.1 only".
  • Tight coupling: Monitoring was baked into infrastructure, not your application code.

What Prometheus Brings to the Table

Service Discovery DYNAMIC

Prometheus integrates natively with Kubernetes, AWS EC2, GCP, Consul, and more. It automatically finds new targets as they appear and drops them when they disappear — no manual config needed.

Pull Model = Safer, Simpler PULL

Applications don't need to know where the monitoring server is. Prometheus finds them. This makes it trivial to test an application locally — just run Prometheus pointed at localhost.

Multi-Dimensional Data Model

Labels give you the power to slice and dice metrics along any dimension without pre-defining aggregations. Change your mind about how to group data? Just write a different PromQL query.

Reliable Alerting via Alertmanager

Prometheus evaluates alert rules locally. The separate Alertmanager handles deduplication, grouping, inhibition, and routing to PagerDuty, Slack, email, and more.

Huge Ecosystem

Hundreds of official and community exporters exist for databases (MySQL, PostgreSQL), message queues (Kafka, RabbitMQ), hardware (node_exporter for Linux), and cloud services.

⚠️ When NOT to use Prometheus

Prometheus is not designed for high-cardinality data (e.g., tracking every individual user ID as a label — that would create millions of time-series). It also stores data locally by default, which limits long-term retention unless you add remote storage like Thanos or Cortex.

The Scrape Cycle — Step by Step

1

Target Discovery

Prometheus reads its configuration file (prometheus.yml) or uses a service discovery mechanism (Kubernetes API, AWS EC2 tags, etc.) to build a list of targets — basically, a list of HTTP endpoints to scrape.

2

HTTP Scrape

At each scrape interval (default: 15 seconds), Prometheus makes an HTTP GET request to each target's /metrics endpoint. The response is a plain text file in the Prometheus exposition format.

3

Parsing & Label Injection

Prometheus parses the metrics, and automatically adds labels like instance (the target's address) and job (the job name from config). You can also relabel metrics during ingestion.

4

Storage in TSDB

Parsed samples are stored in Prometheus's built-in time-series database (TSDB). Data is written to disk in chunks, compressed efficiently, and retained for a configured duration (default: 15 days).

5

Rule Evaluation

Every evaluation interval, Prometheus runs your recording rules (pre-compute expensive queries) and alerting rules (fire alerts when conditions are met). Alert notifications are sent to the Alertmanager.

The Exposition Format

When Prometheus scrapes a target, the target responds with a plain text body that looks like this — it's simple, human-readable, and designed to be efficient to parse:

TEXT Example /metrics endpoint response

The Four Metric Types

Prometheus supports four core metric types. Choosing the right type for your data is important — it determines what operations Prometheus can perform on it.

Counter COUNTER

A value that can only ever go up (or reset to zero on restart). Use it for things you count: requests served, errors occurred, bytes sent. Never use a counter for something that can decrease.

Like the odometer on your car — it only goes forward. Even if you sell the car and the new owner resets it, it still only counts up from there.

http_requests_total{method="GET",status="200"} 45381
Gauge GAUGE

A value that can go up and down arbitrarily. Use it for measurements of current state: memory usage, active connections, queue size, temperature.

Like the fuel gauge on your car — it goes up when you fill the tank, and down as you drive. It reflects the current state at any point in time.

node_memory_MemAvailable_bytes 8589934592
Histogram HISTOGRAM

Samples observations into configurable buckets, plus a running sum and count. Ideal for measuring request durations or response sizes — you want to know what percentage of requests are under 100ms, 500ms, etc.

Imagine a teacher grading 1000 tests and sorting them into piles: "under 50", "50–65", "65–80", "80–100". A histogram gives you this bucketed view of your data distribution.

http_request_duration_seconds_bucket{le="0.1"} 943
Summary SUMMARY

Similar to histograms, but calculates configurable quantiles on the client side (e.g., p50, p95, p99 latency). Summaries are more accurate per-instance but cannot be aggregated across multiple instances like histograms can.

rpc_duration_seconds{quantile="0.95"} 0.00374
💡 Histogram vs Summary — Quick Rule

Use histograms when you want to aggregate across multiple instances (e.g., "what's the p99 latency across all 20 web server pods?"). Use summaries when you need accurate quantiles for a single instance and won't be aggregating.

Setting Up Prometheus on AWS EC2

Let's get Prometheus running on an AWS EC2 instance. We'll install the Prometheus server and the node_exporter, which exposes Linux system metrics (CPU, memory, disk, network).

☁️

AWS EC2 — Prerequisites

Ubuntu 22.04 LTS · t2.micro or larger · Port 9090 open in Security Group

Before you begin, make sure your EC2 Security Group has these inbound rules open:

PortProtocolPurposeSource
22TCPSSH AccessYour IP
9090TCPPrometheus UIYour IP or 0.0.0.0/0
9100TCPNode ExporterPrometheus server IP

Step 1 — Launch and Connect to EC2

BASH SSH into your EC2 instance 
# Connect via SSH (replace with your key and public IP)
ssh -i "your-key.pem" ubuntu@YOUR_EC2_PUBLIC_IP

# Update system packages first
sudo apt-get update -y && sudo apt-get upgrade -y

Step 2 — Install Prometheus

BASH Download and install Prometheus binary 
# Create a dedicated system user for Prometheus (security best practice)
sudo useradd --no-create-home --shell /bin/false prometheus

# Create required directories
sudo mkdir -p /etc/prometheus /var/lib/prometheus

# Download latest Prometheus (check https://prometheus.io/download for latest version)
cd /tmp
wget https://github.com/prometheus/prometheus/releases/download/v2.51.0/prometheus-2.51.0.linux-amd64.tar.gz

# Extract the archive
tar xvf prometheus-2.51.0.linux-amd64.tar.gz
cd prometheus-2.51.0.linux-amd64

# Copy binaries to /usr/local/bin
sudo cp prometheus promtool /usr/local/bin/

# Copy config files and console templates
sudo cp -r consoles console_libraries /etc/prometheus/
sudo cp prometheus.yml /etc/prometheus/

# Set correct ownership
sudo chown -R prometheus:prometheus /etc/prometheus /var/lib/prometheus
sudo chown prometheus:prometheus /usr/local/bin/prometheus /usr/local/bin/promtool

Step 3 — Configure Prometheus

YAML /etc/prometheus/prometheus.yml 
sudo nano /etc/prometheus/prometheus.yml

---
# Global settings apply to all scrape jobs
global:
  scrape_interval:     15s  # How often to scrape targets
  evaluation_interval: 15s  # How often to evaluate rules

# Prometheus scrapes itself — a great sanity check
scrape_configs:

  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  - job_name: 'node_exporter'
    static_configs:
      - targets: ['localhost:9100']
        labels:
          instance: 'aws-ec2-server'
          environment: 'production'

Step 4 — Install Node Exporter

BASH Install node_exporter to expose Linux OS metrics 
# Create user for node_exporter
sudo useradd --no-create-home --shell /bin/false node_exporter

# Download node_exporter
cd /tmp
wget https://github.com/prometheus/node_exporter/releases/download/v1.7.0/node_exporter-1.7.0.linux-amd64.tar.gz

tar xvf node_exporter-1.7.0.linux-amd64.tar.gz
sudo cp node_exporter-1.7.0.linux-amd64/node_exporter /usr/local/bin/
sudo chown node_exporter:node_exporter /usr/local/bin/node_exporter

Step 5 — Create Systemd Services

Creating systemd service files ensures Prometheus and node_exporter start automatically on reboot and restart on failure.

INI /etc/systemd/system/prometheus.service 
sudo nano /etc/systemd/system/prometheus.service

[Unit]
Description=Prometheus Monitoring Server
Wants=network-online.target
After=network-online.target

[Service]
User=prometheus
Group=prometheus
Type=simple
ExecStart=/usr/local/bin/prometheus \
    --config.file=/etc/prometheus/prometheus.yml \
    --storage.tsdb.path=/var/lib/prometheus/ \
    --web.console.templates=/etc/prometheus/consoles \
    --web.console.libraries=/etc/prometheus/console_libraries \
    --storage.tsdb.retention.time=15d \
    --web.enable-lifecycle
Restart=always

[Install]
WantedBy=multi-user.target
INI /etc/systemd/system/node_exporter.service 
sudo nano /etc/systemd/system/node_exporter.service

[Unit]
Description=Node Exporter - Linux System Metrics
Wants=network-online.target
After=network-online.target

[Service]
User=node_exporter
Group=node_exporter
Type=simple
ExecStart=/usr/local/bin/node_exporter
Restart=always

[Install]
WantedBy=multi-user.target
BASH Enable and start both services 
# Reload systemd, enable and start services
sudo systemctl daemon-reload

sudo systemctl enable prometheus node_exporter
sudo systemctl start prometheus node_exporter

# Verify both are running (look for "active (running)")
sudo systemctl status prometheus
sudo systemctl status node_exporter

# Check logs if there's an issue
sudo journalctl -u prometheus -f
sudo journalctl -u node_exporter -f
✅ Verify Access

Open your browser and navigate to http://YOUR_EC2_PUBLIC_IP:9090. You should see the Prometheus web UI. Go to Status → Targets to confirm both prometheus and node_exporter targets show as UP.

Setting Up Prometheus on GCP Virtual Machine

The process on GCP is nearly identical to AWS — the main differences are in how the VM is created and how firewall rules are configured.

🌐

GCP Compute Engine — Prerequisites

Ubuntu 22.04 LTS · e2-micro or larger · Firewall rule for port 9090

Create a VM and configure firewall rules via GCP Console or gcloud CLI.

Step 1 — Create a GCP VM

BASH Create VM using gcloud CLI 
# Authenticate first (if not already)
gcloud auth login

# Create a VM instance
gcloud compute instances create prometheus-server \
  --zone=us-central1-a \
  --machine-type=e2-medium \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud \
  --boot-disk-size=20GB \
  --tags=prometheus-server

# Create firewall rule to allow port 9090 and 9100
gcloud compute firewall-rules create allow-prometheus \
  --allow=tcp:9090,tcp:9100 \
  --target-tags=prometheus-server \
  --source-ranges=0.0.0.0/0 \
  --description="Allow Prometheus and node_exporter access"

# SSH into the VM
gcloud compute ssh prometheus-server --zone=us-central1-a

Step 2 — Install Prometheus and Node Exporter

The installation commands are identical to the AWS steps above (Steps 2–5). Once SSH'd into your GCP VM, run the exact same commands to download binaries, create users, copy files, and set up systemd services.

ℹ️ GCP-Specific Tip

GCP VMs have a metadata server at 169.254.169.254. Prometheus can use GCP service discovery to automatically find all VMs with a specific tag — no static IP configuration needed. This is configured via the gce_sd_configs option in prometheus.yml.

GCP Service Discovery Configuration

Instead of hardcoding IP addresses, Prometheus can dynamically discover all GCP VM instances in a project tagged with prometheus-target:

YAML prometheus.yml — GCP service discovery 
scrape_configs:
  - job_name: 'gcp-vms'
    gce_sd_configs:
      - project: my-gcp-project-id     # Your GCP project ID
        zone: us-central1-a            # Zone to scan
        port: 9100                      # node_exporter port
    relabel_configs:
      - source_labels: [__meta_gce_tag_prometheus]
        action: keep                    # Only keep VMs with this tag
        regex: true
✅ Verify GCP Setup

Navigate to http://YOUR_GCP_EXTERNAL_IP:9090. Check Status → Targets — you should see your targets listed as UP. GCP's external IP can be found in Compute Engine → VM Instances.

Prometheus Built-in HTTP Endpoints

Prometheus ships with a powerful HTTP API. Every Prometheus server — even with zero configuration — exposes these endpoints. Let's explore each one, starting from the most important.

Think of these endpoints as different windows into the same building. The /metrics window shows you all the raw data being collected about the building itself. The /api/v1/query window is like the building's receptionist — you ask questions and get structured answers. The /-/ready window is the fire exit sign — green means the building is safe to use.

GET http://localhost:9090/metrics

The most important endpoint — Prometheus's own metrics page. This is what Prometheus scrapes from itself to monitor its own health. Contains hundreds of metrics about scrape performance, TSDB storage, rule evaluation, and more. We'll explore this in detail in the next section.

GET http://localhost:9090/api/v1/query

The PromQL instant query API. Pass a query parameter ?query=up and Prometheus evaluates the expression at the current moment and returns JSON results. Used by Grafana and all API clients.

curl 'http://localhost:9090/api/v1/query?query=up'
GET http://localhost:9090/api/v1/query_range

Range query API — evaluate a PromQL expression over a time range with a step interval. Returns an array of (timestamp, value) pairs for graphing. Used by Grafana for all graph panels.

curl 'http://localhost:9090/api/v1/query_range?query=rate(http_requests_total[5m])&start=2024-01-01T00:00:00Z&end=2024-01-01T01:00:00Z&step=60s'
GET http://localhost:9090/api/v1/targets

Returns a JSON list of all configured scrape targets with their current state (UP/DOWN), last scrape time, last scrape duration, and labels. Essential for debugging why a target isn't being scraped.

GET http://localhost:9090/api/v1/labels

Returns a list of all label names currently present in the time-series database. Useful for exploring what dimensions of data you have available.

GET http://localhost:9090/api/v1/rules

Returns all configured alerting and recording rules along with their current state. If you have alerting rules defined, this shows which ones are currently firing.

GET http://localhost:9090/-/healthy

Simple health check endpoint — returns HTTP 200 with body Prometheus Server is Healthy. if the server is running. Used by load balancers, Kubernetes liveness probes, and monitoring checks.

GET http://localhost:9090/-/ready

Readiness check — returns HTTP 200 only when Prometheus is ready to serve queries. During startup, this may return 503 while Prometheus loads its TSDB. Used by Kubernetes readiness probes.

POST http://localhost:9090/-/reload

Triggers a live reload of the configuration file. Requires the --web.enable-lifecycle flag on startup. Use this instead of restarting Prometheus when you update prometheus.yml — no data loss, no downtime.

curl -X POST http://localhost:9090/-/reload
GET http://localhost:9090/api/v1/status/tsdb

Returns TSDB (time-series database) statistics: number of series, chunks, samples ingested, head block memory usage. Critical for capacity planning and understanding database growth.

Prometheus Self-Monitoring Metrics

Navigate to http://localhost:9090/metrics and you'll see hundreds of lines of text. This is Prometheus monitoring itself — and it's one of the best ways to learn the Prometheus data format while also understanding how Prometheus actually works internally.

Let's walk through the most important groups of self-metrics you'll encounter:

🔄 Scrape Metrics — "How is data collection going?"

prometheus_target_interval_length_seconds SUMMARY

Actual intervals between scrapes, per target group. If your configured interval is 15s but this shows 30s, something is slow. The quantiles (p0.01, p0.05, p0.5, p0.9, p0.99) show you the distribution of timing variance.

prometheus_target_interval_length_seconds{interval="15s",quantile="0.99"} 15.002
prometheus_target_scrapes_exceeded_sample_limit_total COUNTER

How many times a scrape was rejected because the target returned more samples than the configured sample_limit. A non-zero value here means you're losing data.

prometheus_target_scrapes_exceeded_sample_limit_total 0
prometheus_target_scrapes_sample_duplicate_timestamp_total COUNTER

Counts samples that arrived with a timestamp Prometheus already has. Usually indicates a misconfigured target sending duplicate data.

prometheus_sd_discovered_targets GAUGE

Number of targets currently discovered by each service discovery mechanism. Useful for verifying that service discovery is working — you should see one entry per configured job.

prometheus_sd_discovered_targets{config="node_exporter",name="static"} 1

💾 TSDB Metrics — "How is the database doing?"

prometheus_tsdb_head_series GAUGE

Number of time-series currently in the "head" block (the most recent, in-memory data). This is your current cardinality — the total number of unique metric+label combinations being tracked. Watch this number; unexpected growth can indicate label explosion.

prometheus_tsdb_head_series 1842
prometheus_tsdb_head_chunks GAUGE

Number of chunks in the head block. Each series has its data stored in compressed chunks. More chunks = more memory and disk usage.

prometheus_tsdb_compactions_total COUNTER

Total number of compactions performed. Prometheus periodically compacts older chunks into more efficient blocks on disk. This is a normal background operation.

prometheus_tsdb_storage_blocks_bytes GAUGE

Total size of all persisted blocks on disk in bytes. Use this metric over time to estimate how fast your disk usage grows and plan retention accordingly.

prometheus_tsdb_storage_blocks_bytes 5.4e+08
prometheus_tsdb_wal_corruptions_total COUNTER

Write-Ahead Log corruption count. Should always be zero. A non-zero value indicates disk issues or improper Prometheus shutdown (e.g., kill -9). Prometheus recovers from WAL corruption but may lose some recent data.

🌐 HTTP API Metrics — "How is Prometheus serving requests?"

prometheus_http_requests_total COUNTER

Total HTTP requests served by Prometheus's own HTTP API, broken down by handler path and HTTP status code. Use this to see how heavily your Prometheus API is being queried.

prometheus_http_requests_total{code="200",handler="/api/v1/query"} 4821
prometheus_http_request_duration_seconds HISTOGRAM

Latency histogram for Prometheus's own HTTP API responses. If /api/v1/query is slow, it could mean your PromQL queries are too complex or your TSDB is under load.

⚙️ Rule Evaluation Metrics — "Are alert rules working?"

prometheus_rule_evaluation_duration_seconds SUMMARY

How long rule evaluation takes. If this is slow, your recording or alerting rules are expensive — possibly running heavy PromQL aggregations that could be pre-computed.

prometheus_rule_evaluation_failures_total COUNTER

How many rule evaluations failed. A non-zero count means some of your alerting or recording rules are broken — check logs for the error details.

prometheus_rule_evaluation_failures_total{rule_group="alerting"} 0

⚡ Engine Metrics — "How is PromQL performing?"

prometheus_engine_query_duration_seconds SUMMARY

Duration of PromQL query execution, broken down by query phase (exec_queue_time, inner_eval, result_sort). The inner_eval phase shows your actual query computation time.

prometheus_engine_query_duration_seconds{slice="inner_eval",quantile="0.9"} 0.00423
prometheus_engine_queries_concurrent_max GAUGE

The maximum number of concurrent queries Prometheus allows (default: 20). If you're hitting this limit, queries will be queued and response times will increase.

⚠️ Key Metrics to Always Watch

Set up alerts on: prometheus_tsdb_head_series (cardinality explosion), prometheus_rule_evaluation_failures_total > 0 (broken rules), prometheus_target_scrapes_exceeded_sample_limit_total > 0 (losing data), and up == 0 (any target is down).

Getting Started with PromQL

PromQL (Prometheus Query Language) is how you ask questions of your metrics data. You can run PromQL queries directly in the Prometheus UI at http://localhost:9090/graph. Let's explore the most useful patterns.

PromQL is like SQL — but for time-series data. Instead of tables, you query metric names. Instead of WHERE clauses, you use label selectors in {}. Instead of GROUP BY, you use by() and without(). The core difference is that PromQL is designed for rate-of-change calculations and time windows, not joins.

Pattern 1 — Is everything up?

The up metric is automatically created by Prometheus for each scrape target. A value of 1 means UP, 0 means DOWN.

PROMQL Basic target health check 
# Are all my targets UP?
up

# Show only targets that are DOWN (value = 0)
up == 0

# Show only the node_exporter targets
up{job="node_exporter"}

Pattern 2 — Rate of a Counter

Counters only go up — to calculate how fast they're growing, use rate(). Always apply rate() to counters, never query the raw counter value for graphs.

PROMQL Calculate request rate per second over 5-minute window 
# How many HTTP requests per second right now? (averaged over last 5 min)
rate(prometheus_http_requests_total[5m])

# Same, but only for successful (HTTP 200) requests
rate(prometheus_http_requests_total{code="200"}[5m])

# Total request rate summed across all codes and handlers
sum(rate(prometheus_http_requests_total[5m]))

Pattern 3 — Aggregation Across Labels

PromQL's aggregation operators let you combine multiple time-series into one using mathematical operations.

PROMQL Aggregation with by() and without() 
# Total CPU seconds spent across all modes, per instance
sum by (instance, mode) (rate(node_cpu_seconds_total[5m]))

# CPU utilization percentage (1 - idle)
100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# Available memory in GB
node_memory_MemAvailable_bytes / 1024 / 1024 / 1024

# Error rate as percentage of total requests
rate(http_requests_total{status=~"5.."}[5m])
  /
rate(http_requests_total[5m])
  * 100

Pattern 4 — Prometheus Self-Health Queries

PROMQL Monitor Prometheus itself 
# How many time-series is Prometheus tracking right now?
prometheus_tsdb_head_series

# Scrape success rate (should be near 1.0 for all targets)
avg(up)

# How much disk is the TSDB using? (in GB)
prometheus_tsdb_storage_blocks_bytes / 1024 / 1024 / 1024

# Are any rule evaluations failing?
increase(prometheus_rule_evaluation_failures_total[1h]) > 0

# Average scrape duration across all targets
avg(scrape_duration_seconds) by (job)

Useful PromQL Functions Reference

Function Applies To What It Does Example
rate() Counter Per-second average rate over a window rate(requests_total[5m])
irate() Counter Instant rate (last 2 samples) — spiky, reactive irate(requests_total[5m])
increase() Counter Total increase over a time window increase(errors_total[1h])
avg_over_time() Gauge Average value over a time window avg_over_time(cpu_usage[1h])
histogram_quantile() Histogram Compute quantile from histogram buckets histogram_quantile(0.95, rate(latency_bucket[5m]))
sum() / avg() / max() Any Aggregate across multiple series sum by (job) (up)
count() Any Count the number of series matching count(up == 1)
absent() Any Returns 1 if no series match — great for alerting on missing data absent(up{job="api"})
predict_linear() Gauge Predict future value based on current trend predict_linear(disk_free[1h], 4*3600)
🎯 Golden Rule: Use rate() for counters in graphs. Use raw values for gauges. Use histogram_quantile() for latency percentiles. Use absent() to alert on missing targets.
