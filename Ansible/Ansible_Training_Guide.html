Section A · 25 min
Introduction to Ansible

The Problem: Managing Many Servers Manually

Imagine you just joined a company as a system administrator. The company has 200 web servers, each running Linux. Your manager asks you to install the nginx web server on all 200 of them.

🧑‍💻 Real-World Scenario — Without Ansible
You open 200 terminal windows, SSH into each server one by one, type the same install command 200 times, hope you don't make a typo, and pray every server gets the same configuration. This takes 5–6 hours and is extremely error-prone.

The Pain Points of Manual Work

  • Time-consuming — Logging into 200 machines manually can take an entire workday.
  • Error-prone — A single wrong command on the wrong server can break production.
  • Not repeatable — You must redo every step whenever something changes.
  • Not trackable — Hard to know what was done, by whom, and when.
  • Not scalable — What if you have 2,000 servers next year?

What is Ansible?

Ansible is an open-source IT automation tool. It lets you describe what you want your servers to look like — and then it makes it happen automatically, on all servers at the same time.

DetailInfo
Created byMichael DeHaan, 2012. Acquired by Red Hat in 2015.
LanguageWritten in Python
LicenseOpen Source (GNU GPL)
Config formatYAML (Yet Another Markup Language)
💡 Simple Analogy
Think of Ansible like a recipe book. You write a recipe (called a Playbook) that says "install nginx, start it, and make sure it runs on boot." Ansible reads that recipe and applies it to every server you point it at — simultaneously.

How Ansible Solves the Problem

FeatureWhat it means
AgentlessAnsible needs NO software installed on target servers. It connects over SSH. Nothing to maintain on managed machines.
IdempotentRun the same playbook 10 times — it only makes changes when needed. If nginx is already installed, Ansible skips it.
DeclarativeYou describe the desired end-state ("nginx should be installed"), not step-by-step instructions. Ansible figures out the steps.
ParallelAnsible can run tasks on 200 servers simultaneously, not one by one.
Human-readablePlaybooks are written in YAML — a format that reads almost like English sentences.
✅ With Ansible — Same Scenario
You write ONE playbook file. Run ONE command. Ansible SSHs into all 200 servers in parallel, installs nginx, and reports back with a full success/failure summary. This takes 2 minutes instead of 6 hours.

Installing Ansible
⚠️ Important Note
Ansible can only be installed on Linux or macOS. On Windows, you must use WSL (Windows Subsystem for Linux), which gives you a Linux environment inside Windows.

Installing on Windows (via WSL)

Step 1 — Enable WSL

PowerShell (as Administrator)
# Run this in PowerShell as Administrator
wsl --install

# After restart, WSL installs Ubuntu by default.
# Open "Ubuntu" from the Start Menu to get a Linux terminal.

Step 2 — Install Ansible inside WSL (Ubuntu)

Ubuntu Terminal (inside WSL)
# Update package list
sudo apt update

# Install required tool
sudo apt install software-properties-common -y

# Add Ansible's official repository
sudo add-apt-repository --yes --update ppa:ansible/ansible

# Install Ansible
sudo apt install ansible -y

# Verify installation
ansible --version

Installing on Ubuntu / Debian Linux

Terminal
sudo apt update
sudo apt install software-properties-common -y
sudo add-apt-repository --yes --update ppa:ansible/ansible
sudo apt install ansible -y
ansible --version

Installing on macOS

macOS Terminal
# Step 1: Install Homebrew (if not already installed)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Step 2: Install Ansible
brew install ansible

# Step 3: Verify
ansible --version

Expected Output

Expected terminal output
ansible [core 2.16.0]
  config file = /etc/ansible/ansible.cfg
  configured module search path = ['~/.ansible/plugins/modules']
  python version = 3.11.4
  jinja version = 3.1.2
  libyaml = True
✅ You're Ready!
If you see the output above, Ansible is successfully installed on your machine. Your machine is now the Control Node.

Ansible Architecture
🏗️ Architecture in One Sentence
The Control Node reads a Playbook and an Inventory file, then connects to the Managed Nodes via SSH to execute tasks.

The Three Core Concepts

1 — Control Node

Definition: The machine where Ansible is installed. This is the machine YOU run commands from.

  • This is your laptop or a dedicated automation server.
  • Ansible is ONLY installed here — not on the target servers.
  • It reads playbooks and inventory files from here.
  • Analogy: The chef's kitchen. The chef (Ansible) prepares and sends instructions from here.

2 — Managed Nodes (Target Machines)

Definition: The servers that Ansible manages. Ansible SSHs into these to run tasks.

  • No Ansible installation needed here — just Python and SSH.
  • Could be web servers, database servers, AWS EC2 instances, Azure VMs, etc.
  • Analogy: The restaurant branches. The head chef sends instructions; the branches follow them.

3 — Inventory

Definition: A file listing all Managed Nodes (their IP addresses or hostnames) that Ansible should manage.

  • Think of it as your "address book" of servers.
  • Can be a simple INI text file or a YAML file.
  • Servers can be grouped (e.g., [webservers], [databases]).
  • Analogy: A staff directory. The Control Node needs to know where to connect.

Inventory — Deep Dive + Hands-On Demo

An inventory file is a plain text file that tells Ansible which machines to manage. It contains IP addresses or hostnames of your servers. Think of it as the "contacts list" that Ansible dials up.

📋 Demo Context
In this demo, we will use an AWS EC2 instance as our Managed Node. You will need the public IP of a running EC2 instance and the SSH private key (.pem file) used to launch it.

Hands-On Demo: Ping Test to AWS EC2

Prerequisites

  • AWS EC2 instance running (Amazon Linux 2 or Ubuntu)
  • Your .pem private key file (e.g., my-key.pem)
  • The EC2 public IP (e.g., 13.233.45.67)
  • Port 22 (SSH) open in the EC2 Security Group
  • Ansible installed on your machine
Create Project Directory

Set up your working folder and secure your key file.

Create Inventory File

Write an INI-format inventory listing your EC2 IP.

Run Ansible Ping

Execute ansible all -i inventory.ini -m ping.

Verify "pong" Response

SUCCESS + "pong" confirms Ansible can connect.

Terminal — Step by Step
# Step 1: Create project folder
mkdir ansible-demo && cd ansible-demo

# Step 2: Copy and secure your .pem key
cp ~/Downloads/my-key.pem .
chmod 400 my-key.pem

# Step 3: Create the inventory file
cat > inventory.ini << EOF
[webservers]
13.233.45.67  ansible_user=ec2-user  ansible_ssh_private_key_file=./my-key.pem
EOF

# Step 4: Run the ping test
ansible webservers -i inventory.ini -m ping

Expected Output (Success)

Terminal output
13.233.45.67 | SUCCESS => {
    "ansible_facts": {
        "discovered_interpreter_python": "/usr/bin/python3"
    },
    "changed": false,
    "ping": "pong"
}
🎉 What Just Happened?
Ansible read your inventory.ini, found the IP, SSHed into it using your .pem key, ran a small Python script on the remote machine, and returned "pong" — confirming the connection works. You never logged into the server manually!

Troubleshooting Common Issues

ErrorFix
UNREACHABLE — port 22 timed outCheck EC2 Security Group: inbound rule for port 22 from 0.0.0.0/0 must exist.
Permission denied (publickey)Check ansible_ssh_private_key_file path is correct. Ensure chmod 400 was run.
Wrong user errorAmazon Linux uses "ec2-user"; Ubuntu uses "ubuntu"; CentOS uses "centos".
UNPROTECTED PRIVATE KEYRun: chmod 400 my-key.pem

INI vs YAML Inventory Formats

INI Format (Simple, Beginner-Friendly)

inventory.ini
# Group: webservers
[webservers]
13.233.45.67  ansible_user=ec2-user  ansible_ssh_private_key_file=./my-key.pem
13.233.45.68  ansible_user=ec2-user  ansible_ssh_private_key_file=./my-key.pem

# Group: databases
[databases]
10.0.0.5  ansible_user=ubuntu  ansible_ssh_private_key_file=./db-key.pem

# Group of groups
[production:children]
webservers
databases

YAML Format (Advanced, More Flexible)

inventory.yml
all:
  children:
    webservers:
      hosts:
        13.233.45.67:
          ansible_user: ec2-user
          ansible_ssh_private_key_file: ./my-key.pem
        13.233.45.68:
          ansible_user: ec2-user
          ansible_ssh_private_key_file: ./my-key.pem
    databases:
      hosts:
        10.0.0.5:
          ansible_user: ubuntu
          ansible_ssh_private_key_file: ./db-key.pem
Use INI when...Use YAML when...
You are a beginnerYou have complex nested group hierarchies
Small number of servers (< 50)You need host-level variables per server
Simple, flat inventoryBuilding dynamic inventory from scripts
Quick readability mattersConsistency with playbooks (all YAML)

Section E · 25 min
Playbook, Play, Task & Module
📖 Recipe Book Analogy
Think of a Playbook as a recipe book. Inside it, there are Plays (recipes for a specific meal). Each Play has Tasks (individual cooking steps). Each Task uses a Module (a specific kitchen tool — like a knife, pan, or oven).

Module — The Smallest Unit

A Module is the smallest unit of work in Ansible. It is a reusable, built-in tool that does one specific thing.

ModuleWhat it does
pingTests if Ansible can connect to a host.
apt / yumInstalls or removes packages (apt for Ubuntu/Debian, yum for CentOS/RHEL).
copyCopies a file from Control Node to Managed Node.
serviceStarts, stops, or restarts a service (nginx, apache, mysql, etc.).
fileCreates, deletes, or sets permissions on files and directories.
userCreates or manages Linux user accounts.
shell / commandRuns a raw shell command on the target server.
templateCopies a file with variables substituted (Jinja2 templates).

Task — A Single Step

A Task is a single instruction that calls a Module with specific parameters.

Example Task
# Install nginx using the "apt" module
- name: Install nginx web server
  apt:
    name: nginx
    state: present

Play — Map Tasks to Hosts

A Play maps a group of Tasks to a group of hosts from your Inventory. A Play says: "Run these tasks on these servers."

Example Play
- name: Set up web servers
  hosts: webservers
  become: yes
  tasks:
    - name: Install nginx
      apt:
        name: nginx
        state: present

    - name: Start nginx service
      service:
        name: nginx
        state: started

Playbook — The Full Recipe

A Playbook is a YAML file that contains one or more Plays. It is the top-level file you run with the ansible-playbook command.

setup-webserver.yml ← This is the PLAYBOOK
---
# Play 1: Web servers
- name: Set up web servers
  hosts: webservers
  become: yes
  tasks:
    - name: Install nginx
      apt:
        name: nginx
        state: present

# Play 2: Database servers
- name: Set up database servers
  hosts: databases
  become: yes
  tasks:
    - name: Install MySQL
      apt:
        name: mysql-server
        state: present

Visual Hierarchy

Hierarchy Overview
PLAYBOOK  (setup-webserver.yml)
└── PLAY 1  (targets: webservers)
│     ├── TASK 1: Install nginx   [uses: apt module]
│     ├── TASK 2: Start nginx     [uses: service module]
│     └── TASK 3: Copy config     [uses: copy module]
│
└── PLAY 2  (targets: databases)
      ├── TASK 1: Install MySQL   [uses: apt module]
      └── TASK 2: Start MySQL     [uses: service module]

Section F · 40 min
Writing & Running Your First Playbook
🎯 Goal
By the end of this section, you will have written a real Ansible Playbook, run it against an EC2 instance, and seen it install and start a web server — automatically!

What We Will Build

A Playbook that does the following on an Ubuntu EC2 instance:

  1. Updates the package list (like running apt update manually)
  2. Installs the nginx web server
  3. Starts the nginx service
  4. Enables nginx to start automatically on server reboot
  5. Creates a custom index.html page to verify it worked

Step-by-Step Guide

Step 1 — Create the Playbook File

Terminal
cd ansible-demo
nano setup-webserver.yml

Step 2 — Write the Playbook

setup-webserver.yml
---
# Playbook: setup-webserver.yml
# Purpose: Install and configure nginx on Ubuntu EC2

- name: Set up Nginx Web Server
  hosts: webservers
  become: yes

  tasks:

    - name: Update apt package list
      apt:
        update_cache: yes

    - name: Install nginx
      apt:
        name: nginx
        state: present

    - name: Start and enable nginx service
      service:
        name: nginx
        state: started
        enabled: yes

    - name: Create custom index.html
      copy:
        content: "<h1>Hello from Ansible! Server: {{ inventory_hostname }}</h1>"
        dest: /var/www/html/index.html
        mode: "0644"

Step 3 — Check Syntax

Terminal
ansible-playbook -i inventory.ini setup-webserver.yml --syntax-check

# Expected output:
playbook: setup-webserver.yml

Step 4 — Dry Run (Check Mode)

Terminal
# Simulate without making real changes
ansible-playbook -i inventory.ini setup-webserver.yml --check

Step 5 — Run the Playbook

Terminal
ansible-playbook -i inventory.ini setup-webserver.yml

# Add -v for verbose output
ansible-playbook -i inventory.ini setup-webserver.yml -v

Understanding the Output

Expected Output

TASK [Gathering Facts] *************************************
ok: [13.233.45.67]

TASK [Update apt package list] *****************************
changed: [13.233.45.67]

TASK [Install nginx] ***************************************

TASK [Start and enable nginx service] **********************

StatusMeaning
okTask ran, nothing changed. Desired state was already in place.
changedTask ran and made a change on the server.
failedTask encountered an error. Playbook stops here by default.
skippedTask was skipped due to a condition (when: clause) not being met.
unreachableAnsible could not connect to the server. Check SSH and inventory.
🏆 Congratulations!
You just automated a server setup with Ansible! Without it, this would have required manually SSH-ing into the server and running 4–5 commands. With Ansible, it was one command — and it works just as easily on 200 servers.
✅ Idempotency in Action
Run the same playbook again. You'll see changed=0. Nginx is already installed and running, so Ansible does nothing. This is idempotency — safe to run multiple times without side effects.
