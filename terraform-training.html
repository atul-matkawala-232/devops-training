01

Why Terraform?

⏱ ~20 min

Before Terraform, setting up cloud infrastructure was a painful, manual process — one wrong click in the AWS console and your production environment could look completely different from staging. Terraform changed all that.

🏠 Analogy — The Blueprint

Imagine you're building a house. You wouldn't tell each worker verbally what to do each morning. Instead, you'd hand them a blueprint. Everyone reads the same blueprint, builds the same house, every time. Terraform is the blueprint for your cloud infrastructure.

📝

Declarative Syntax

You describe what you want, not how to do it. Terraform figures out the steps.

☁️

Multi-Cloud Support

One tool to rule AWS, GCP, Azure, Kubernetes and 3000+ providers using the same HCL language.

🔄

Idempotent

Run the same config 10 times — you get the same result. No duplicates, no surprises.

👥

Team-Friendly

Infrastructure lives in Git. Teams collaborate, review, and audit changes like code.

🆓

Open Source

Core Terraform (now OpenTofu) is free. Large community, rich ecosystem of modules.

📊

Plan Before Apply

See exactly what will change before anything is modified. No more surprise deletions.

The IaC Philosophy

Infrastructure as Code (IaC) means treating your servers, networks, databases, and cloud resources the same way you treat application code — versioned, reviewed, automated, and repeatable.

Before IaCAfter IaC with Terraform
Click around in AWS ConsoleWrite a .tf file, run terraform apply
Document steps in a Word docThe code IS the documentation
"Works on my account"Same config, identical infra every time
Rebuild from memory if something breaksRe-run Terraform to restore instantly
No audit trail of changesFull git history of every infrastructure change

What Problem Does Terraform Solve?

⏱ ~20 min

The Core Problems

😱

Configuration Drift

Over time, manually managed environments diverge. Dev looks different from Prod. Terraform enforces consistency.

⏳

Slow Provisioning

Spinning up 10 VMs manually takes hours. Terraform automates it in minutes with a single command.

🔍

No Visibility

Who created that S3 bucket? When? Why? With Terraform + Git, every change has an author and timestamp.

🔄

Hard to Replicate Environments

Creating identical dev/staging/prod environments was a nightmare. Terraform workspaces and modules solve this.

☕ Analogy — Coffee Shop Recipe

Imagine a barista making coffee by memory vs. following a precise recipe card. The recipe card (Terraform config) ensures every customer gets the same cup, whether made in Mumbai or New York. Manual console work is the "by memory" approach — inconsistent and human-error prone.

Terraform vs Alternatives

ToolTypeBest ForKey Difference
TerraformIaCMulti-cloud infra provisioningDeclarative, state management, huge ecosystem
AnsibleConfig MgmtConfiguring existing serversAgentless, procedural, better for app config
AWS CloudFormationIaCAWS-onlyNative AWS, JSON/YAML, no state file needed
PulumiIaCDevs who prefer real languagesUses Python/JS/Go instead of HCL
AWS CDKIaCAWS + programming language fansCode compiles to CloudFormation
💡

Key insight: Terraform doesn't replace Ansible. They're complementary — Terraform provisions the infrastructure (VMs, networks, buckets), Ansible configures what runs on it (install nginx, set up users, deploy apps).

Architecture of Terraform

⏱ ~25 min

Understanding how Terraform works internally helps you debug issues faster and design better configurations.

Core Components Explained

1. Terraform Core

The brain of Terraform. It reads your configuration files, consults the state file to understand what already exists, builds a dependency graph, and creates an execution plan. Written in Go, it communicates with providers via a gRPC protocol.

2. Providers

Providers are plugins that know how to talk to a specific cloud or service API. When you write provider "aws" {}, Terraform downloads the AWS provider plugin that wraps all AWS API calls.

🔌 Analogy — Universal Adaptor

Think of Terraform Core as a laptop, and providers as power adaptors. The laptop (Core) is the same everywhere, but you need a specific adaptor (provider) to plug into India's sockets (AWS), UK sockets (GCP), or US sockets (Azure).

3. State File (terraform.tfstate)

Terraform maintains a JSON file that maps your HCL configuration to real cloud resources. This is how it knows "this resource block = that EC2 instance ID." Without state, Terraform would have no memory of what it created.

4. The Execution Flow

  1. Write — You write HCL configs describing desired infrastructure
  2. Init — terraform init downloads providers and sets up the backend
  3. Plan — Terraform compares your config to current state and shows a diff
  4. Apply — Terraform executes the plan, calls cloud APIs, updates state
  5. State Sync — terraform.tfstate is updated to reflect real world

Resource Graph

Terraform builds a DAG (Directed Acyclic Graph) of all resources. Resources without dependencies are provisioned in parallel; those that depend on others wait. This makes Terraform fast and correct — it always creates a VPC before subnets that live inside it.

⚠️

Critical: The state file is sensitive — it may contain passwords, secrets, and resource IDs. Never commit terraform.tfstate to Git. Use remote backends (S3, GCS) with encryption.

Key Real-World Components

📁

Git Repository

All .tf files live in Git. Every infrastructure change goes through a pull request — with review, history, and rollback ability.

🔐

Secrets Management

Cloud credentials stored in GitHub Actions secrets, HashiCorp Vault, or AWS IAM roles — never hardcoded in .tf files.

🗄️

Remote State

State stored in S3 or GCS with DynamoDB/GCS locking to prevent two people applying changes simultaneously.

🔁

CI/CD Integration

GitHub Actions or GitLab CI runs terraform plan on PRs and terraform apply on merges.

🏦 Analogy — Bank Approval Process

Think of a home loan approval. You submit an application (PR with .tf files). A reviewer checks it (terraform plan in CI). Management approves (PR approval). The bank executes the loan (terraform apply). There's a complete audit trail of who approved what and when — just like Terraform in a proper DevOps pipeline.

Terraform Setup in Real-World Teams

⏱ ~20 min

Enterprise-Grade Setup Checklist

  • Remote state backend (S3 + DynamoDB for AWS, GCS for GCP)
  • State locking enabled (prevents concurrent applies)
  • State encryption at rest
  • Terraform version pinned in required_version
  • Provider versions locked in .terraform.lock.hcl
  • Separate state files per environment (dev/staging/prod)
  • CI/CD pipeline with plan-on-PR, apply-on-merge
  • Secrets via environment variables or Vault, never in .tf files

Backend Configuration — AWS (S3)

backend.tf
terraform {
  backend "s3" {
    bucket         = "my-company-terraform-state"
    key            = "prod/networking/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true

    # DynamoDB table for state locking
    dynamodb_table = "terraform-state-lock"
  }

  required_version = ">= 1.5.0"
}

Fundamentals of Terraform (HCL)

⏱ ~30 min

HCL (HashiCorp Configuration Language) is Terraform's language. It's designed to be human-readable and machine-parseable. If you've seen JSON or YAML, you'll pick it up quickly.

Core Building Blocks

1. Providers — Connect to Cloud

main.tf
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
  # Credentials come from env vars or AWS CLI config
}

2. Resources — The Core Unit

A resource block defines a real-world infrastructure object you want to create.

main.tf
# Syntax: resource "provider_type" "local_name" { ... }
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-bucket-2024"

  tags = {
    Environment = "dev"
    Team        = "platform"
  }
}

# Referencing another resource: resource_type.local_name.attribute
resource "aws_s3_bucket_versioning" "versioning" {
  bucket = aws_s3_bucket.my_bucket.id
  versioning_configuration {
    status = "Enabled"
  }
}

3. Data Sources — Read Existing Resources

main.tf
# Data sources read existing infrastructure — they don't create anything
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

# Reference: data.aws_ami.amazon_linux.id

4. Variables — Make Configs Reusable

variables.tf
variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.micro"
}

variable "environment" {
  description = "Deployment environment"
  type        = string
  # No default = required input
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Must be dev, staging, or prod."
  }
}

5. Outputs — Expose Values

outputs.tf
output "bucket_arn" {
  description = "ARN of the created S3 bucket"
  value       = aws_s3_bucket.my_bucket.arn
}

output "bucket_name" {
  value = aws_s3_bucket.my_bucket.bucket
}

6. Locals — Computed Values

main.tf
locals {
  common_tags = {
    Environment = var.environment
    Project     = "terraform-demo"
    ManagedBy   = "terraform"
  }
  # Use as local.common_tags throughout your config
}

HCL Types Reference

TypeExampleWhen to Use
string"us-east-1"Region names, IDs, labels
number3Port numbers, counts
booltrueFlags, toggles
list(string)["a", "b"]Multiple AZs, CIDR blocks
map(string){key="val"}Tags, environment variables
objectComplex nestedGrouped settings

Working with Terraform

⏱ ~35 min

The Core Workflow: Init → Plan → Apply → Destroy

🔧

terraform init

Downloads providers, sets up backend, initializes modules. Always run first. Safe to run multiple times.

🔍

terraform plan

Shows what will change: + add, ~ modify, - destroy. No changes made. Use -out plan.tfplan to save.

🚀

terraform apply

Executes the plan. Asks for confirmation. Use -auto-approve in CI/CD. Can use saved plan file.

💥

terraform destroy

Destroys all managed resources. Same as apply -destroy. Useful for teardown of demo environments.

✈️ Analogy — Flight Checklist

init is pre-flight checks (load fuel, check systems). plan is the flight path shown on screen (here's where we're going). apply is taking off (executing). destroy is landing and decommissioning the aircraft. You'd never skip the pre-flight — never skip plan in production!

Plan Output Reading

# Legend:
+ resource   # Will be CREATED
~ resource   # Will be MODIFIED in-place
- resource   # Will be DESTROYED
-/+ resource # Will be REPLACED (destroy + create)

# Example output:
  # aws_s3_bucket.my_bucket will be created
  + resource "aws_s3_bucket" "my_bucket" {
      + bucket = "my-unique-bucket-2024"
      + id     = (known after apply)
    }

Plan: 1 to add, 0 to change, 0 to destroy.

Variables in Practice

terraform apply -var="environment=prod" -var="instance_type=t3.large"
terraform.tfvars (auto-loaded)
environment   = "prod"
instance_type = "t3.large"
region        = "us-east-1"
prod.tfvars (pass explicitly)
terraform apply -var-file="prod.tfvars"
# Prefix with TF_VAR_ — Terraform auto-reads these
export TF_VAR_environment="prod"
export TF_VAR_instance_type="t3.large"
terraform apply

Breaking main.tf Into Multiple Files

Terraform reads ALL .tf files in a directory. It's a convention — not a requirement — to organize code into separate files:

FilePurpose
main.tfCore resources (what you're creating)
variables.tfAll input variable declarations
outputs.tfAll output declarations
providers.tfProvider and terraform blocks
locals.tfLocal values and computed expressions
backend.tfRemote state backend configuration
terraform.tfvarsVariable values (auto-loaded)

Terraform Registry

The Terraform Registry at registry.terraform.io is the official source for providers and public modules. Think of it like npm for JavaScript or PyPI for Python — but for infrastructure.

Terraform Console

An interactive REPL for testing expressions without modifying infrastructure:

bash

"Hello, prod"

Terraform Modules

⏱ ~30 min
🧩 Analogy — LEGO Bricks

Modules are like LEGO sets. Instead of building a castle brick-by-brick every time, you create a "castle kit" (module) with all the right pieces, and just assemble new castles by calling that kit. Once you've built a VPC module, every team can use it to create consistent, correct VPCs without knowing the internals.

What Are Modules?

A module is any directory containing .tf files. Every Terraform configuration is technically a module (the root module). You create child modules to encapsulate and reuse groups of resources.

Creating Your First Module

modules/
└── s3_bucket/
    ├── main.tf        # The bucket resource
    ├── variables.tf   # Inputs to the module
    └── outputs.tf     # What the module exposes
modules/s3_bucket/variables.tf
variable "bucket_name" {
  description = "Name of the S3 bucket"
  type        = string
}

variable "environment" {
  type    = string
  default = "dev"
}
modules/s3_bucket/main.tf
resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name
  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}
modules/s3_bucket/outputs.tf
output "bucket_id" {
  value = aws_s3_bucket.this.id
}
output "bucket_arn" {
  value = aws_s3_bucket.this.arn
}

Using Your Module

root/main.tf
module "logs_bucket" {
  source      = "./modules/s3_bucket"
  bucket_name = "my-app-logs-2024"
  environment = "prod"
}

module "assets_bucket" {
  source      = "./modules/s3_bucket"
  bucket_name = "my-app-assets-2024"
  environment = "prod"
}

# Access outputs: module.logs_bucket.bucket_arn

Using Public Modules

The Terraform Registry has thousands of battle-tested modules. Example: HashiCorp's official VPC module:

main.tf
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"

  name = "my-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["us-east-1a", "us-east-1b"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]

  enable_nat_gateway = true
  single_nat_gateway = true
}
🎯

Best practice: Always pin public module versions with version = "5.0.0". Never use a floating version — a module update could break your infrastructure on next terraform init.

Root Module Structure

Your root module is the entry point — it stitches together child modules, sets provider configs, and declares the backend. Think of it as the orchestrator.

main.tf — EC2 Lab
# Look up the latest Amazon Linux 2023 AMI dynamically

  name        = "terraform-demo-sg"
  description = "Allow SSH and HTTP"

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]  # Demo only — restrict in production!

  ingress {

Estimated time: 20 minutes
main.tf — GCP VM
# VPC Network
resource "google_compute_network" "demo" {
  name                    = "terraform-demo-vpc"
  auto_create_subnetworks = false
}

# Subnet
resource "google_compute_subnetwork" "demo" {
  name          = "terraform-demo-subnet"
  ip_cidr_range = "10.10.0.0/24"
  region        = var.region
  network       = google_compute_network.demo.id
}

# Firewall rule — allow SSH and HTTP
resource "google_compute_firewall" "allow_ssh_http" {
  name    = "allow-ssh-http"
  network = google_compute_network.demo.name

  allow { protocol = "tcp"; ports = ["22", "80"] }
  source_ranges = ["0.0.0.0/0"]
  target_tags   = ["web-server"]
}

# Compute Engine VM
resource "google_compute_instance" "web" {
  name         = "terraform-demo-vm"
  machine_type = var.machine_type
  zone         = var.zone
  tags         = ["web-server"]

  boot_disk {
    initialize_params {
      image = "debian-cloud/debian-12"
      size  = 20
    }
  }

  network_interface {
    subnetwork = google_compute_subnetwork.demo.id
    access_config {}  # Gives a public IP
  }

  metadata_startup_script = <<-EOF
    #!/bin/bash
    apt-get update
    apt-get install -y apache2
    echo "<h1>VM Deployed by Terraform on GCP!</h1>" > /var/www/html/index.html
    systemctl enable apache2
    systemctl start apache2
  EOF

  labels = {
    environment = "terraform-lab"
    managed_by  = "terraform"
  }
}
outputs.tf — GCP VM
output "vm_name"       { value = google_compute_instance.web.name }
output "external_ip"   { value = google_compute_instance.web.network_interface[0].access_config[0].nat_ip }
output "website_url"   { value = "http://${google_compute_instance.web.network_interface[0].access_config[0].nat_ip}" }

Lab 2 — Create a Cloud Storage Bucket (GCS)
Estimated time: 15 minutes · A core DevOps service for artifacts, logs & backups

GCS (Google Cloud Storage) is the equivalent of AWS S3 — used heavily in DevOps for storing build artifacts, Terraform state, application logs, and backups.

gcs.tf
resource "google_storage_bucket" "artifacts" {
  name          = "${var.project_id}-terraform-artifacts"
  location      = "US"
  force_destroy = true

  # Versioning — like S3 versioning
  versioning {
    enabled = true
  }

  # Lifecycle policy — auto-delete old versions after 30 days
  lifecycle_rule {
    condition {
      num_newer_versions = 3
      age                = 30
    }
    action { type = "Delete" }
  }

  # Uniform bucket-level access (replaces per-object ACLs)
  uniform_bucket_level_access = true

  labels = {
    environment = "terraform-lab"
    managed_by  = "terraform"
  }
}

# IAM binding — give the VM service account read access to the bucket
resource "google_storage_bucket_iam_member" "vm_reader" {
  bucket = google_storage_bucket.artifacts.name
  role   = "roles/storage.objectViewer"
  member = "serviceAccount:${var.project_id}@appspot.gserviceaccount.com"
}
outputs.tf — GCS
output "bucket_name" { value = google_storage_bucket.artifacts.name }
output "bucket_url"  { value = google_storage_bucket.artifacts.url }
output "self_link"   { value = google_storage_bucket.artifacts.self_link }

Run It

terraform init
terraform plan -var="project_id=your-gcp-project-id"
terraform apply -var="project_id=your-gcp-project-id"
terraform destroy -var="project_id=your-gcp-project-id"

Terraform State

⏱ ~25 min
🧾 Analogy — Inventory Register

State is like a shop's inventory register. If you have 10 laptops on shelf but your register says 8, your record is wrong — and you might order 2 more unnecessarily. Terraform's state file is its register of what it believes exists in your cloud. If the state diverges from reality, Terraform makes wrong decisions.

What Does State Contain?

The state file (terraform.tfstate) is a JSON file mapping each resource block to the real cloud resource's ID and all its attributes.

// terraform.tfstate (simplified)
{
  "version": 4,
  "resources": [
    {
      "type": "aws_s3_bucket",
      "name": "demo",
      "instances": [
        {
          "attributes": {
            "id":     "my-unique-bucket-2024",
            "arn":    "arn:aws:s3:::my-unique-bucket-2024",
            "region": "us-east-1",
            ...
          }
        }
      ]
    }
  ]
}

Remote State

For teams, state must be stored remotely so everyone shares the same source of truth:

backend.tf — AWS S3 Backend
terraform {
  backend "s3" {
    bucket         = "company-terraform-state"
    key            = "prod/app/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"  # Prevents concurrent applies
  }
}
backend.tf — GCS Backend
terraform {
  backend "gcs" {
    bucket = "company-terraform-state"
    prefix = "prod/app"
  }
}

State Commands

CommandPurpose
terraform state listList all resources tracked in state
terraform state show aws_instance.webShow all attributes of a specific resource
terraform state mv old_name new_nameRename/move resource in state (after refactoring)
terraform state rm aws_s3_bucket.oldRemove from state (stop managing, don't destroy)
terraform import aws_s3_bucket.existing bucket-idImport an existing resource into state
terraform refreshSync state with real-world (deprecated; use apply -refresh-only)

State Locking

When two engineers run terraform apply simultaneously, they could corrupt the state file. State locking prevents this — only one apply can run at a time. AWS uses DynamoDB for locking, GCP uses built-in GCS locking.

Workspaces

Workspaces let you manage multiple environments (dev/staging/prod) from the same configuration with separate state files:

terraform workspace new dev
terraform workspace new prod
terraform workspace select prod
terraform workspace list
⚠️

Workspaces vs separate directories: Many teams prefer separate directories per environment over workspaces, as it makes configuration differences explicit and prevents accidental cross-environment changes.

Terraform Best Practices

⏱ ~20 min
📁

Use Consistent File Structure

Always split into main.tf, variables.tf, outputs.tf, providers.tf. Readable code is maintainable code.

🔒

Never Hardcode Secrets

No credentials, passwords, or API keys in .tf files. Use environment variables, Vault, or cloud secret managers.

📌

Pin Provider & Module Versions

Use exact or range-constrained versions. Commit .terraform.lock.hcl to Git to lock provider checksums.

🏷️

Tag All Resources

Use locals for consistent tags across all resources: Environment, ManagedBy, Team, Project, CostCenter.

🗄️

Remote State Always

Local state is fine for learning; use S3/GCS + locking for any real environment. Never commit tfstate to Git.

🔍

Plan Before Every Apply

Always review the plan output. In CI/CD, post the plan as a PR comment. Never apply without understanding what changes.

🧩

Use Modules for Reuse

Any pattern you create more than twice should become a module. This enforces consistency across teams and projects.

✅

Run terraform fmt and validate

Always format before committing. Add terraform validate as a pre-commit hook or CI step.

🛡️

Use Least Privilege IAM

Terraform should run with minimal permissions. Use separate service accounts/roles per environment.

The Golden Rules (Summary)

  1. Always init first — Run terraform init whenever you clone a repo or add a new provider
  2. Always plan before apply — Read the plan carefully, especially for -/+ (replace) operations
  3. State is sacred — Store remotely, lock it, encrypt it, back it up
  4. Secrets stay out — Infrastructure is open; credentials are not
  5. Use version control — Every infra change via Git PR, reviewed, traceable
  6. Modules for everything repeated — DRY (Don't Repeat Yourself) applies to infrastructure too
