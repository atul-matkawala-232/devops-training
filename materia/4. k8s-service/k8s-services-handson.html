Why Services?
The core problem they solve

Pods are temporary. Every time a pod dies and restarts it gets a new IP address. A Service gives you a stable, permanent endpoint that always routes to healthy pods — no matter how many there are or what their IPs are.

With a Service

Setup Steps
Complete these before moving on
1
Enable Kubernetes in Docker Desktop

Docker Desktop → ⚙️ Settings → Kubernetes → Enable Kubernetes → Apply & Restart

Wait ~2 minutes for the green "Kubernetes is running" dot at the bottom of Docker Desktop.

2
Verify your cluster context
kubectl config current-context
docker-desktop
⚠️ Wrong context?Run: kubectl config use-context docker-desktop
3
Create a clean namespace
kubectl create namespace svc-demo
kubectl config set-context --current --namespace=svc-demo
4
Create working folder & save the ReplicaSet YAML
mkdir ~/k8s-svc-demo && cd ~/k8s-svc-demo

cat > app-replicaset.yaml << 'EOF'
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
  namespace: svc-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
EOF
5
Apply the ReplicaSet & wait for pods to be Running
kubectl apply -f app-replicaset.yaml

# Watch pods come up (Ctrl+C when all 3 show Running)
kubectl get pods -w

nginx-rs-def34 1/1 Running 0 9s

ClusterIP
Internal-only communication between pods
📌 What is ClusterIP? ClusterIP is the default Service type. It creates a virtual IP address reachable only from inside the cluster. Perfect for pod-to-pod communication — frontend talking to backend, app talking to database.

Traffic Flow — ClusterIP

Frontend Pod
inside cluster
──→
ClusterIP
10.96.x.x:80
stable IP
──→
Pod 1
Pod 2
Pod 3

🚫 Browser / external traffic cannot reach ClusterIP — this is intentional

clusterip-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-clusterip
  namespace: svc-demo
spec:
  type: ClusterIP        # Default — you can omit this line
  selector:
    app: nginx            # Matches any pod labelled app=nginx
  ports:
  - port: 80             # Port the Service listens on (inside cluster)
    targetPort: 80       # Port on the pod container to forward to
1
Save the YAML file
cat > clusterip-service.yaml << 'EOF'
apiVersion: v1
kind: Service
metadata:
  name: nginx-clusterip
  namespace: svc-demo
spec:
  type: ClusterIP
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
EOF
2
Apply and inspect
kubectl apply -f clusterip-service.yaml
kubectl get svc
NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE
nginx-clusterip ClusterIP 10.96.45.123 <none> 80/TCP 5s
Notice EXTERNAL-IP is <none> — correct! ClusterIP has no external access.
Experiment A — Prove it routes to pods (from inside the cluster)
# Run a temporary busybox pod inside the cluster and wget the Service
kubectl run test-pod --image=busybox --restart=Never -it --rm \
  -- wget -qO- http://nginx-clusterip:80

# You will see the nginx HTML! Run it again — you may hit a different pod.
Experiment B — Confirm it is NOT reachable from your Mac Try opening http://localhost:80 in your browser. It will not connect. ClusterIP is invisible from outside — this is exactly correct behaviour. ✅
Experiment C — Check pod details & labels
# See all pods with their labels
kubectl get pods --show-labels

# Describe the Service — see Endpoints section (pod IPs it routes to)
kubectl describe svc nginx-clusterip

NodePort
External access via a port on the node (localhost on Docker Desktop)
📌 What is NodePort? NodePort builds on top of ClusterIP and additionally opens a specific port (30000–32767) on every node. On Docker Desktop, your node IS localhost — so hitting localhost:30080 in a browser reaches your pods.

Traffic Flow — NodePort

🌐 Browser
localhost:30080
──→
NodePort
:30080
external port
──→
Pod 1
Pod 2
Pod 3
🔢 Port Terminology
port: 80 — port the Service listens on inside the cluster
targetPort: 80 — port on the pod container to forward to
nodePort: 30080 — the external port opened on the node (must be 30000–32767)
nodeport-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-nodeport
  namespace: svc-demo
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
  - port: 80             # internal cluster port
    targetPort: 80       # pod container port
    nodePort: 30080      # external port on node — must be 30000-32767
1
Save the YAML file
cat > nodeport-service.yaml << 'EOF'
apiVersion: v1
kind: Service
metadata:
  name: nginx-nodeport
  namespace: svc-demo
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080
EOF
2
Apply and inspect
kubectl apply -f nodeport-service.yaml
kubectl get svc
NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE
nginx-clusterip ClusterIP 10.96.45.123 <none> 80/TCP 2m
nginx-nodeport NodePort 10.96.78.200 <none> 80:30080/TCP 5s
PORT(S) shows 80:30080 — internal port 80 mapped to external port 30080. Open your browser at http://localhost:30080 now!
Experiment A — Access from your browser Open http://localhost:30080 in Safari or Chrome. You should see the nginx welcome page! Traffic went: browser → localhost:30080 → NodePort → one of the 3 pods.
Experiment B — See load balancing in action
# First get your pod names
kubectl get pods

# Give each pod a unique homepage
kubectl exec <pod-1-name> -- bash -c "echo '<h1>Pod 1 is serving you!</h1>' > /usr/share/nginx/html/index.html"
kubectl exec <pod-2-name> -- bash -c "echo '<h1>Pod 2 is serving you!</h1>' > /usr/share/nginx/html/index.html"
kubectl exec <pod-3-name> -- bash -c "echo '<h1>Pod 3 is serving you!</h1>' > /usr/share/nginx/html/index.html"

# Now refresh http://localhost:30080 several times
# You'll see Pod 1, Pod 2, Pod 3 in rotation — load balancing!
Experiment C — Self-healing through the Service (two terminals)
# Terminal 1 — continuous curl to show traffic
while true; do curl -s http://localhost:30080 | grep "<h1>"; sleep 0.5; done

# Terminal 2 — kill a pod and watch Terminal 1
kubectl delete pod <any-pod-name>

# Terminal 1 keeps running — Service routes around the dead pod.
# ReplicaSet creates a new pod, Service picks it up automatically.

LoadBalancer
Production-grade external access with a clean IP/hostname
📌 What is LoadBalancer? LoadBalancer builds on top of NodePort and provisions a real external load balancer. On cloud clusters it gets a dedicated IP (e.g. 34.120.x.x). On Docker Desktop it assigns localhost — giving you clean port 80 access with no NodePort number needed.

Traffic Flow — LoadBalancer

🌐 Browser
localhost:80
──→
LoadBalancer
External IP
cloud or localhost
──→
Pod 1
Pod 2
Pod 3
loadbalancer-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-loadbalancer
  namespace: svc-demo
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80             # Clean external port — no nodePort field needed
    targetPort: 80       # Kubernetes auto-assigns a NodePort behind the scenes
1
Save and apply the YAML
cat > loadbalancer-service.yaml << 'EOF'
apiVersion: v1
kind: Service
metadata:
  name: nginx-loadbalancer
  namespace: svc-demo
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
EOF

kubectl apply -f loadbalancer-service.yaml
2
Watch for EXTERNAL-IP to appear
# -w watches for changes — wait a few seconds
kubectl get svc -w
NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE
nginx-clusterip ClusterIP 10.96.45.x <none> 80/TCP 5m
nginx-nodeport NodePort 10.96.78.x <none> 80:30080/TCP 3m
nginx-loadbalancer LoadBalancer 10.96.90.x <pending> 80/TCP 2s
nginx-loadbalancer LoadBalancer 10.96.90.x localhost 80/TCP 8s
EXTERNAL-IP changes from <pending> to localhost — Docker Desktop assigned it! On a cloud cluster this would be a real IP like 34.120.x.x
3
Open in browser — clean port 80!

Go to http://localhost — no port number needed. This is how production apps feel to end users.

open http://localhost
Experiment A — View all three Services together This single command tells the whole story of the three Service types side by side. 
kubectl get svc
NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S)

nginx-nodeport NodePort 10.96.78.200 <none> 80:30080/TCP

🏗️ Services Stack on Each Other Every LoadBalancer includes a NodePort, which includes a ClusterIP. Creating a LoadBalancer gives you all three capabilities at once. That's why kubectl describe svc shows a NodePort even on a LoadBalancer.

Clean Up
Remove all demo resources from your cluster
✅ Great work completing the hands-on! Deleting the namespace removes everything inside it — the ReplicaSet, all 3 pods, and all 3 Services — in one command.
1
Delete the entire demo namespace (removes everything)
kubectl delete namespace svc-demo
namespace "svc-demo" deleted
2
Reset default namespace back to default
kubectl config set-context --current --namespace=default
3
Verify everything is gone
kubectl get all -n svc-demo
# Expected: No resources found in svc-demo namespace.
🎯 What's Next? — Deployments Now that you understand Services, the natural next topic is Deployments.

A Deployment manages a ReplicaSet and adds:
  • Rolling updates — update pods one-by-one, zero downtime
  • Rollback — instantly revert if something breaks
  • Update history — Kubernetes keeps old ReplicaSets for rollback

Deployments + Services together are the core of how real apps run in production Kubernetes.
