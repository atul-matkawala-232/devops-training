Core Concepts — What Are We Building?

Architecture Overview — What You'll Build

🏗️ VPC (Virtual Private Cloud)

Think of a VPC as your own private data center inside AWS. It's an isolated section of the AWS cloud where you control everything — IP ranges, subnets, routing, and security. Nothing comes in or out unless you explicitly allow it.

📦 Subnets

Subnets are smaller divisions inside your VPC. Think of floors in a building — public subnets face the internet (like a lobby), while private subnets are hidden (like secure offices). Resources placed in different subnets are naturally isolated from each other.

🗺️ Route Tables

Route tables are traffic maps for your network. They tell AWS: "If traffic is going to X destination, send it via Y gateway." Every subnet must be associated with a route table. Without proper routes, your resources can't communicate — even within the same VPC.

🌐 Internet Gateway (IGW)

The Internet Gateway is the door between your VPC and the public internet. Without it, nothing in your VPC can talk to the outside world (and vice versa). You attach one IGW per VPC, then add a route pointing to it in your public subnet's route table.

🔗 VPC Peering

VPC Peering lets two VPCs talk to each other privately — as if they're on the same network. Traffic stays on the AWS backbone (never touches the public internet). Useful when you have separate VPCs for different teams or environments that need to share resources.

LAB 01

Create Your First VPC

💡
What we're building:

A VPC named MyLabVPC with CIDR block 10.0.0.0/16 — giving us 65,536 possible IP addresses to work with.

01

Navigate to the VPC Dashboard

Log in to the AWS Console. In the search bar at the top, type VPC and click on the VPC service. You'll land on the VPC Dashboard.

02

Click "Create VPC"

On the left sidebar click Your VPCs, then the orange Create VPC button in the top-right.

03

Fill in the VPC Settings

Configure the following fields:

FieldValueWhy?
Resources to createVPC onlyWe'll create subnets manually
Name tagMyLabVPCIdentifies your VPC
IPv4 CIDR10.0.0.0/16Gives us 65K IPs to divide
TenancyDefaultShared hardware (cheaper)
04

Create via AWS CLI (Alternative)

If you prefer the command line, here's the equivalent command:

bash — AWS CLI 
# Create the VPC
aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16 \
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=MyLabVPC}]'

# The output will include your new VpcId, e.g.: vpc-0abc12345678
05

Verify It's Created

Your new VPC will appear in the Your VPCs list with state Available. Note down the VPC ID — you'll need it in the next labs.

✅
Lab 1 Complete!

You now have an isolated private network on AWS. It can't communicate with anything yet — that's intentional. Next, we'll divide it into subnets.

LAB 02

Create Public & Private Subnets

💡
What we're building:

Two subnets inside MyLabVPC — one public (10.0.1.0/24) for internet-facing resources like web servers, and one private (10.0.2.0/24) for internal resources like databases.

Subnet CIDR Breakdown
01

Open the Subnets section

In the VPC Dashboard left sidebar, click Subnets, then click Create subnet.

02

Create the Public Subnet

FieldValue
VPC IDSelect MyLabVPC
Subnet namePublicSubnet
Availability Zoneus-east-1a (any AZ)
IPv4 CIDR10.0.1.0/24
03

Add a second subnet — Private Subnet

Click Add new subnet in the same dialog (or repeat the process) and fill in:

FieldValue
Subnet namePrivateSubnet
Availability Zoneus-east-1b (different AZ for HA)
IPv4 CIDR10.0.2.0/24
04

CLI equivalent

bash — AWS CLI 
# Replace vpc-XXXXXX with your actual VPC ID
VPC_ID=vpc-XXXXXX

# Create Public Subnet
aws ec2 create-subnet \
  --vpc-id $VPC_ID \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=PublicSubnet}]'

# Create Private Subnet
aws ec2 create-subnet \
  --vpc-id $VPC_ID \
  --cidr-block 10.0.2.0/24 \
  --availability-zone us-east-1b \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=PrivateSubnet}]'
05

Enable Auto-assign Public IP on PublicSubnet

Select PublicSubnet → Actions → Edit subnet settings → check Enable auto-assign public IPv4 address. This ensures EC2 instances launched here get a public IP automatically.

⚠️
Naming ≠ Behavior:

Calling a subnet "Public" doesn't make it public! It only becomes truly public after you attach an Internet Gateway and add a route to it (Labs 3 & 4).

LAB 03

Configure Route Tables

💡
What we're building:

A dedicated Public Route Table for PublicSubnet and leaving the Main Route Table for PrivateSubnet. Each will have different routing rules.

01

Create the Public Route Table

In the VPC Dashboard, click Route Tables → Create route table:

FieldValue
NamePublicRouteTable
VPCMyLabVPC
02

Associate PublicSubnet with this Route Table

Select PublicRouteTable → Subnet associations tab → Edit subnet associations → check PublicSubnet → Save.

03

Understand the Default Route (Local)

Every route table gets a local route by default — it lets all resources within the same VPC communicate freely. You'll see it listed as:

DestinationTargetMeaning
10.0.0.0/16localAll VPC traffic stays inside
04

We'll Add the Internet Route in Lab 4

Once the Internet Gateway is created, we'll come back here and add the route 0.0.0.0/0 → IGW. Think of 0.0.0.0/0 as "everything else" — the default route for internet-bound traffic.

05

CLI equivalent

bash — AWS CLI 
# Create a new route table
aws ec2 create-route-table \
  --vpc-id $VPC_ID \
  --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=PublicRouteTable}]'

# Save the RouteTableId from output, e.g.: rtb-0abc123456
RT_ID=rtb-XXXXXX

# Associate with Public Subnet
aws ec2 associate-route-table \
  --route-table-id $RT_ID \
  --subnet-id $PUBLIC_SUBNET_ID

LAB 04

Attach an Internet Gateway

💡
Two-step process:

Creating an IGW is not enough. You must (1) create the IGW, (2) attach it to your VPC, and (3) add a route in the public route table pointing to it.

01

Create the Internet Gateway

In VPC Dashboard → Internet Gateways → Create internet gateway:

FieldValue
Name tagMyLabIGW

After creation, notice the state is Detached — it's floating, not connected to anything yet.

02

Attach IGW to MyLabVPC

Select MyLabIGW → Actions → Attach to VPC → choose MyLabVPC → Attach. The state changes to Attached.

⚠️
One VPC, one IGW.

A VPC can only have one Internet Gateway attached at a time.

03

Add the Internet Route to PublicRouteTable

Go back to Route Tables → select PublicRouteTable → Routes tab → Edit routes → Add route:

DestinationTargetMeaning
10.0.0.0/16localVPC internal (auto-added)
0.0.0.0/0igw-XXXXXXAll internet traffic → IGW
04

CLI equivalent (all three steps)

bash — AWS CLI 
# Step 1: Create IGW
aws ec2 create-internet-gateway \
  --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=MyLabIGW}]'

IGW_ID=igw-XXXXXX   # from output

# Step 2: Attach to VPC
aws ec2 attach-internet-gateway \
  --internet-gateway-id $IGW_ID \
  --vpc-id $VPC_ID

# Step 3: Add route (0.0.0.0/0 → IGW) to Public Route Table
aws ec2 create-route \
  --route-table-id $RT_ID \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id $IGW_ID
05

Verify — Can Resources Reach the Internet?

Launch a test EC2 instance in PublicSubnet. Assign a Security Group that allows inbound SSH (port 22) and outbound all traffic. SSH in and try:

bash — EC2 Instance 
# From inside your EC2 instance:
ping google.com

# Expected output: 64 bytes from ... time=X ms ✅
# If it fails — check Security Group outbound rules first
✅
Lab 4 Complete!

Your public subnet can now reach the internet. Your private subnet still has no internet access — that's the correct secure architecture. For private subnet internet access, you'd add a NAT Gateway (not covered in this lab).

LAB 05

VPC Peering — Connect Two VPCs

💡
Scenario:

You have MyLabVPC (10.0.0.0/16) and want to connect it with a second VPC VPC-B (172.16.0.0/16). Resources in both VPCs should be able to communicate using private IPs.

VPC Peering — 4-Step Setup
01

Create VPC-B (the second VPC)

Repeat Lab 1 to create a second VPC. Use a non-overlapping CIDR — this is critical!

FieldValue
NameVPC-B
CIDR172.16.0.0/16
⚠️
Non-overlapping CIDRs required.

If VPC-A is 10.0.0.0/16, VPC-B cannot use any range that overlaps — like 10.0.1.0/24. Use a completely different range like 172.16.0.0/16.

02

Request the Peering Connection (from VPC-A)

In VPC Dashboard → Peering connections → Create peering connection:

FieldValue
NameVPC-A-to-B-Peering
Requester VPCMyLabVPC (VPC-A)
Accepter accountMy account (same account)
Accepter VPCVPC-B
03

Accept the Peering Request (from VPC-B side)

The peering connection shows as Pending Acceptance. Select it → Actions → Accept request. It becomes Active.

04

Update Route Table in VPC-A

Go to VPC-A's route table → Add a route to reach VPC-B via the peering connection:

DestinationTarget
172.16.0.0/16pcx-XXXXXX (Peering Connection)
05

Update Route Table in VPC-B

Now go to VPC-B's route table and add the reverse route — so VPC-B can reply back to VPC-A. Peering is not transitive — you must add routes on both sides:

DestinationTarget
10.0.0.0/16pcx-XXXXXX (same Peering Connection)
06

CLI equivalent

bash — AWS CLI 
# Create peering connection request
aws ec2 create-vpc-peering-connection \
  --vpc-id $VPC_A_ID \
  --peer-vpc-id $VPC_B_ID

PCX_ID=pcx-XXXXXX   # from output

# Accept the peering connection
aws ec2 accept-vpc-peering-connection \
  --vpc-peering-connection-id $PCX_ID

# Add route in VPC-A's route table → VPC-B CIDR via peering
aws ec2 create-route \
  --route-table-id $VPC_A_RT_ID \
  --destination-cidr-block 172.16.0.0/16 \
  --vpc-peering-connection-id $PCX_ID

# Add route in VPC-B's route table → VPC-A CIDR via peering
aws ec2 create-route \
  --route-table-id $VPC_B_RT_ID \
  --destination-cidr-block 10.0.0.0/16 \
  --vpc-peering-connection-id $PCX_ID
07

Test the Peering Connection

Launch an EC2 in each VPC. From the instance in VPC-A, ping the private IP of the instance in VPC-B:

bash — EC2 in VPC-A 
# Replace with the actual private IP of EC2 in VPC-B
ping 172.16.1.50

# If it fails, check Security Groups on both instances
# The Security Group of VPC-B instance must allow ICMP from 10.0.0.0/16
✅
Lab 5 Complete!

The two VPCs are now privately connected. Traffic between them never leaves the AWS network. Remember: peering is not transitive — if you add VPC-C, you'd need separate peering connections (A↔C and B↔C).

🚨 Common Mistakes to Avoid

  • ✗
    Overlapping CIDRs in VPC Peering — VPC-A: 10.0.0.0/16 and VPC-B: 10.0.1.0/24 overlap, peering will fail.
  • ✗
    Missing the route after attaching IGW — Attaching the IGW is not enough; you MUST add 0.0.0.0/0 → IGW in the route table.
  • ✗
    One-sided peering routes — Adding the route only in VPC-A means VPC-A can send but VPC-B can't respond. Add routes on both sides.
  • ✗
    Security Groups blocking traffic — Even with correct routing, Security Groups can block traffic. Always check SG inbound rules too.
  • ✗
    Forgetting to associate subnet with route table — Subnets default to the Main Route Table. Explicitly associate them with your custom route table.
